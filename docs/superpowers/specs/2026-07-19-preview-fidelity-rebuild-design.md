# MC Server Manager 预览保真重建设计

## 决策与目标

采用已确认的 A 方案：以用户批准的视觉预览作为逐屏视觉合同，重建实际 React/Electron 应用，而不是继续在旧页面结构上追加主题变量和局部 CSS。

本规格取代此前 `2026-07-19-overworld-studio-full-app-redesign-design.md` 中过于宽泛的视觉实施部分。此前已确认的功能、安全和数据要求继续有效；若视觉描述冲突，以本规格为准。

目标：

- 实际应用在结构、比例、密度、组件尺寸和视觉层级上接近批准预览。
- 保留服务器创建、管理、市场、Java、设置和日志的真实功能。
- 同时完成深色、浅色与跟随系统主题。
- 排除用户明确否决的编辑推荐/`Adventure Together` 横幅。
- 消除当前“旧页面结构 + 新主题颜色 + 局部卡片”的混合界面。

## 非目标

- 不修改数据库、Electron command、服务端安装、进程管理或 EULA 后端契约。
- 不新增 CurseForge API 密钥。
- 不伪造 BBSMC 关注数、图片质量、服务端包验证或可安装性。
- 不以重设计为理由重写 React、Electron、TanStack Query、Zustand 或 Radix UI。

## 视觉合同

### 应用外壳

- 保留原生窗口标题栏、完整左侧栏、主内容区和底部状态栏。
- 展开侧栏具有稳定宽度、清晰品牌区、一级导航和服务器状态；折叠侧栏仍保留可辨识图标与状态，不缩成近乎不可见的细线。
- 主内容区使用稳定页面内边距和明确画布边界；宽屏内容不会无限拉伸，也不会因固定高度留下大片空白。
- 全局侧栏仅包含服务器、Java 运行时、应用日志和设置。市场仅在创建服务器和编辑模组包服务器上下文出现。

### 排版与尺寸

- 页面标题约 `28px`，区块标题 `16–18px`，正文 `13–14px`，数据和辅助信息 `11–12px`。
- 输入框目标高度约 `40px`，常规按钮目标高度约 `36px`，图标、文字和热区保持一致。
- 页面、工具栏、卡片、列表行和详情面板共享统一的间距级别，不沿用旧版全局超紧凑尺寸。
- 标题、路径、版本名、远程文本和错误堆栈必须显式换行、省略或内部滚动，不得裁切到相邻区域。

### 颜色与 Minecraft 辨识度

- 使用低饱和苔藓绿、石板色和少量金色；绿色用于主要操作和可用状态，金色用于克制的强调与提醒。
- 辨识度来自方块几何、地图工作台式分区、状态符号和克制的地形图形。
- 不使用像素字体、大面积游戏贴图、高饱和霓虹、通用蓝紫渐变或推荐横幅。
- 浅色和深色主题保持相同结构与状态含义，跟随系统模式监听系统主题变化。

### 共享组件

- 页面标题、工具栏、工作区面板、侧栏菜单、数据卡片、结果卡、空状态、错误状态和状态徽章使用统一结构。
- 表面层级依靠背景、细边框和排版建立；阴影只用于窗口、浮层和需要明确层级的元素。
- 交互状态必须覆盖 hover、pressed、disabled 与 `focus-visible`；鼠标点击不保留键盘焦点环。
- 加载状态保持最终布局尺寸，避免骨架或进度出现时引发布局跳动。

## 页面合同

### 1. 外壳与服务器仪表盘

- 仪表盘使用批准预览的工作台比例，展示服务器状态摘要、服务器集合、运行时健康和近期活动。
- 服务器身份、状态、版本和主要操作具有稳定位置；卡片和列表视图共享同一尺寸系统。
- 无服务器时提供明确的拖入/选择服务端包和创建服务器入口。

### 2. 服务器详情与控制台

- 服务器身份、Minecraft/加载器版本、端口、内存和启动操作固定在工作区顶部。
- 左侧菜单承载控制台、文件、内容、备份、设置和活动，不使用将所有内容堆在单页的方式。
- 控制台由主日志区和健康侧栏组成；低宽度下侧栏移动到日志下方。
- 命令建议展示命令、说明和参数格式，支持方向键、Tab、Enter 与 Escape，并明确来源是内置命令目录。

### 3. 创建服务器流程

- 六步进度固定在主内容区头部：来源、兼容性、Java、服务器配置、检查与 EULA、安装并启动。
- 来源页保持大面积拖放区与清晰的四种入口，并在选择市场后直接进入上下文市场。
- Java 与服务器设置作为向导主内容步骤，不使用弹窗。
- 未安装时离开显示放弃确认；安装开始后任务进入后台并可恢复。
- EULA 必须由用户主动打开并确认，不能默认勾选。

### 4. 市场与整合包详情

- 市场筛选和结果使用批准预览的工作台结构，不显示推荐横幅。
- 结果卡显示标题、作者、简介、来源、下载量、Minecraft、加载器、服务端可用性和明确操作。
- 低分辨率封面限制为方形媒体位并完整显示，不拉伸或裁成横幅；不得伪造高分辨率图片。
- 结果始终使用批准预览中的单列宽卡片；窄屏时缩小媒体位并让操作区换行，不改成另一套卡片结构。
- 项目详情保留在主内容区，包含项目身份、兼容性、画廊、正文和服务端版本侧栏。
- BBSMC 公共直链可选择；外部网盘版本仅说明手动下载并禁用直接安装；缺失关注数据不显示 `0`。

### 5. Java、设置与应用日志

- Java 页面使用完整主内容工作区展示发现的运行时、兼容性、占用服务器、路径和来源。
- 设置使用主内容区左侧二级菜单，分类内容在右侧切换；外观页面提供浅色、深色和跟随系统预览。
- 应用日志使用筛选侧栏、事件列表和详情面板的三级结构。宽屏列表获得主要宽度，窄屏按可读顺序堆叠。
- 重复日志按级别、来源、消息和堆栈签名分组；长消息在列表省略，完整内容在详情显示。

## 样式与组件架构

当前单一 `src/styles.css` 已形成旧规则和后加覆盖交叠的问题。重建采用四层样式边界：

1. 主题令牌：颜色、排版、间距、半径、尺寸和状态。
2. 应用外壳：标题栏、侧栏、状态栏、主画布和响应式断点。
3. 共享组件：按钮、表单、面板、卡片、列表、菜单和状态。
4. 页面布局：仪表盘、服务器、向导、市场、Java、设置和日志。

实施时可以继续由 `src/styles.css` 作为入口，但必须按上述边界拆分到职责清晰的样式文件，删除被替代的旧规则，不保留依赖 CSS 顺序碰巧覆盖的重复选择器。

React 组件按页面合同调整语义结构；不通过伪元素或绝对定位模拟缺失的结构。AppShell 只管理全局导航，页面状态保持在现有页面组件和 store 中。

## 数据流与错误处理

- 继续通过 TanStack Query 调用现有 Electron preload command。
- 查询和 mutation 错误在当前工作区内展示原因、重试和日志入口，不静默降级。
- 后台任务继续使用现有创建生命周期；离开页面不能取消已开始任务或伪造完成。
- 远程 Markdown、图片代理和外部链接继续使用现有安全边界。
- 视觉组件不得改变提交 payload、数据库字段或服务器进程行为。

## 分批实施与验收

### 批次 1：基础与主工作台

- 主题令牌、共享尺寸、应用外壳、侧栏、页面画布和服务器仪表盘。

### 批次 2：服务器工作区

- 服务器详情、左侧菜单、控制台、健康侧栏、命令建议、文件与内容入口。

### 批次 3：创建与市场

- 创建向导、来源、市场结果、项目详情、Java/配置步骤、EULA 和安装进度。

### 批次 4：系统页面与状态

- Java、设置、应用日志、加载、空白、失败、危险确认和最终响应式收敛。

每个批次必须在开始下一批前完成：

- 深色与浅色主题。
- `1280×720`、`1440×900`、`1920×1080` 三种视口。
- 与批准预览并排截图审查。
- 无水平页面滚动、文字裁切、无意义底部空白或不可访问的嵌套滚动。
- 相关组件测试、样式合同测试、完整 Vitest 和生产构建。

批次 4 完成后运行 Electron 后端冒烟和 UI 冒烟测试。

## 实施规则

- 每个批次先建立能捕获当前差距的失败测试，再实现最小结构和样式。
- 不在一个批次中顺手重构无关业务逻辑。
- 不以测试通过替代视觉验收；测试和截图均为完成条件。
- 不接受只替换颜色、只追加覆盖规则或仅修改一张示例页面作为完成。
- 工作区已有改动必须保留；提交时按批次精确限定文件和差异。
